# Brainstorming 强制工作流规范

> **文档版本**: 1.0
> **创建日期**: 2025-12-27
> **适用范围**: ContentRSS 项目及所有新功能开发

---

## 1. 核心原则

> **在写代码之前，必须完成需求梳理和设计确认。**

这不是建议，这是强制规则。

---

## 2. 触发条件

当用户说以下任何一句时，**必须**先启动 brainstorming 流程：

- "我想做一个 XX 功能"
- "帮我加一个 YY"
- "能不能实现 ZZ"
- "我有个想法..."
- 任何涉及新功能、新组件、行为修改的请求

---

## 3. Brainstorming 流程

### Step 1: 理解上下文
- 检查项目文件、文档、最近 commits
- 理解现有架构和约束

### Step 2: 一次一个问题
- **优先多选题**，比开放题更容易回答
- 每条消息只问一个问题
- 聚焦：目的、约束、成功标准

### Step 3: 探索 2-3 个方案
- 展示权衡
- 给出推荐并说明理由

### Step 4: 分段呈现设计
- 每段 200-300 词
- 每段结束后确认："这样对吗？"
- 覆盖：架构、组件、数据流、错误处理、测试

### Step 5: 输出设计文档
- 保存到 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 提交到 Git

### Step 6: 创建实施计划
- 拆分为 2-5 分钟的小任务
- 每个任务有明确的文件路径、代码、验证步骤

---

## 4. YAGNI 原则

> **You Aren't Gonna Need It**

在设计阶段，强制问自己：
- 这个功能是必须的吗？
- 能不能先做最小版本？
- 哪些可以延后？

---

## 5. 禁止的行为

| ❌ 禁止 | ✅ 正确做法 |
|:---|:---|
| 直接开始写代码 | 先完成 brainstorming |
| 一次问多个问题 | 一次一个 |
| 跳过设计确认 | 每段设计都要用户确认 |
| 边做边改需求 | 先冻结设计，再执行 |

---

## 6. 与 Superpowers 的集成

如果 Superpowers 已安装，使用以下命令：

```bash
/superpowers:brainstorm    # 启动 brainstorming
/superpowers:write-plan    # 创建实施计划
/superpowers:execute-plan  # 分批执行
```
